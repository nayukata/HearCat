# hearcat

Mac の音声をリアルタイムに日本語文字起こし・録音し、その会話ログをこの AI が読んで質疑応答するためのツール。実体はメニューバー常駐アプリ(HearCat.app)で、CLI `hearcat` はアプリと会話ログへの唯一の窓口。話者は `自分`(マイク)と `相手`(システム音声)の2種類。文字起こしは `[時刻] 話者: 本文` の形式で追記され続ける。

## 絶対に守る原則

**HearCat の会話ログには、built-in ファイル操作ツール(Read / Write / Edit / Bash の cat, head, tail, sed, awk, grep, echo, mv, rm など)を絶対に使わない**。読み書きは常に `hearcat` CLI 経由。`~/Library/Application Support/HearCat/sessions/` 配下のファイルを、CLI を介さずに開かない・書かない・移動しない・消さない。

理由:

- 原文 `<セッション名>.md` は音声認識の一次記録。上書きされると復元できない
- 保存階層は将来変わり得る。パス直書きの手順はすぐ壊れる
- 権限・同時アクセスの制御を CLI が担っている

CLI に無い操作でどうしても必要なら、無理に代替せず「その操作は CLI にありません」とユーザーへ返す。

## 最初にやること(重要)

このスキルが起動した時点で、返答を組み立てる前に必ず `hearcat status` を実行し、セッションが動いていれば続けて `hearcat read --tail 200` で直近の文字起こしを把握する。ユーザーが明確な質問をしていなくても、以下のような発話は「会話ログを参照した上で返答してほしい」というサインとして扱う:

- 状況報告: 「今MTG中」「打ち合わせ中」「面談してる」「通話中」等 → セッションが動いていなければ「今から始めますか」と確認、動いていれば黙って直近の文脈を掴んで待機(ユーザーの次の一言に文脈込みで答えられる状態にする)。
- 過去参照: 「さっき」「あの件」「今の話」等 → `hearcat read` の内容を根拠に回答。文字起こしに該当箇所が無ければ「文字起こしには残っていない」と正直に言う。憶測で埋めない。

セッションが動いていない状態で「今MTG中」と言われた場合は、`hearcat start` を提案する。ユーザーが黙認したら自動で始めるのではなく、必ず一言ことわる(マイク許可のダイアログが出る可能性があるため)。

## 前提

- macOS 26 以降。`hearcat` CLI が PATH 上にあり、HearCat.app が導入済みであること(未導入なら HearCat.app の設定 → AI エージェント連携から導入するか、リポジトリの `install.sh` を実行する)。
- 初回のセッション開始時に、マイク・音声認識の許可ダイアログが出る。利用者に許可してもらう。
- 初回は日本語(ja-JP)の認識モデルのダウンロードが走ることがある。

## セッションの制御

セッションを開始する(アプリが未起動なら自動で起動する。録音と文字起こしの両方がオンで始まる):

```sh
hearcat start
```

録音せず文字起こしだけ、のように片方を切って始めることもできる:

```sh
hearcat start --no-record       # 文字起こしのみ
hearcat start --no-transcribe   # 録音のみ
```

セッション途中で片方だけ切り替える:

```sh
hearcat set record off      # 録音だけ止める(文字起こしは続く)
hearcat set transcribe on   # 文字起こしを再開する
```

セッションを停止する(末尾の発話を確定し、録音ファイルを閉じてから終了する):

```sh
hearcat stop
```

状態を確認する(セッション中か、録音/文字起こしのオンオフ、どのファイルに書いているか):

```sh
hearcat status
```

最新の文字起こしファイルのパスを得る(表示用の参考。読み書きは `hearcat read` / `hearcat write-cleaned` を使うので普段は使わない):

```sh
hearcat latest
```

## 会話ログの読み方

**必ず `hearcat read` を使う**。built-in の Read や `cat` は使わない。

最新セッション(録音中を含む直近)の原文全体:

```sh
hearcat read
```

直近 N 行だけ:

```sh
hearcat read --tail 200
```

過去のセッション(セッション ID を指定):

```sh
hearcat read 2026-07-08_024527
hearcat read 2026-07-08_024527 --tail 500
```

要約が生成済みなら:

```sh
hearcat read --summary                 # 最新
hearcat read 2026-07-08_024527 --summary
```

清書(hearcat-clean スキルが生成する) が生成済みなら:

```sh
hearcat read --cleaned
```

**読んだ内容だけを根拠に答える**。文字起こしに載っていないことは「ログには無い」と言う。`自分` と `相手` のラベルは音声経路の別なので、`相手` 側に複数人いても区別されない点は忘れずに。

セッション中でもファイルは随時追記されるので、その時点までの内容を読める。ただし直近数秒の発話は暫定扱いで未確定=未書き込みなので、「今まさに言われた一言」は取れないことがある。

## 過去のセッションを探す

「昨日のMTG」「先週の Atnd の話」など過去への言及があったら、`hearcat sessions` でセッション一覧を得る。TSV(タブ区切り: `id \t 開始日時ISO8601 \t セッション名 \t フォルダ`)で並び順は新しい順:

```sh
hearcat sessions
```

フォルダで絞る(プロジェクトフォルダで整理してある場合):

```sh
hearcat sessions --folder Atnd
```

該当セッションが決まったら:

1. 要約があれば `hearcat read <id> --summary` で先に読む(全体像が早くつかめる)
2. 本文を確認するなら `hearcat read <id>` (全体)または `hearcat read <id> --tail 500` (末尾)
3. 検索したいキーワードがある場合、まずは `hearcat sessions` でセッションを絞ってから `hearcat read` してから該当行を目視で探す(現状 CLI に文字列検索は無い)

## 要約・議事録を頼まれたら

「議事録作って」「今の話をまとめて」「サマリー出して」等の依頼:

1. 対象セッションを決める(明示が無ければ現在進行中/直近)。
2. `hearcat read <id> --summary` を試し、生成済みならそれを AI 側で議事録形式(参加者・議題・決定事項・宿題・次回)へ整形する。無ければ `hearcat read <id>` の本文から直接構成する。
3. **アプリ側の要約生成 (Apple Intelligence) は CLI から呼べない**。ユーザーがアプリで要約を作りたい場合は「メニューバーの猫アイコン → 履歴 → 対象セッションを右クリック → 要約生成」を案内する。
4. 議事録はテキストで返す。ファイルへ書き出したい依頼が来ても、**セッションディレクトリ内へは書かない**。書きたいなら別のパス(デスクトップ等)を指定してもらう。

## 清書(誤変換の修正)を頼まれたら

「清書して」「文字起こしを整えて」「誤変換を直して」等の依頼は、別スキル `hearcat-clean` の担当。そちらを起動する(呼び出し元のシステムで hearcat-clean が有効なら自動で切り替わる)。

**このスキルからは絶対に清書結果を書き込まない**。原文 `<セッション名>.md` は一次記録なので、上書き・書き換えを絶対にしない。清書結果を書き戻すなら hearcat-clean 側の手順に従って `hearcat write-cleaned` を使う。

## CLI から触れないこと(アプリ側の操作を案内する)

以下は CLI に無いので、`メニューバーの猫アイコン` からの操作を案内する:

- ホットキー変更・録音音量調整・カレンダー連携のオンオフ → 「パネル → 設定」
- 過去セッションの再生・名前変更・フォルダ移動・削除 → 「パネル → 履歴 → 右クリック」
- ライブの暫定テキスト(未確定発話)の閲覧 → 「パネル → 履歴 → ライブ」
- Apple Intelligence による要約生成 → 「履歴 → セッション右クリック → 要約生成」(Apple Silicon + Apple Intelligence 有効時のみ)
- AI エージェント連携(スキル・CLI)の再導入 → 「設定 → AI エージェント連携 → 導入する」

## 既知の挙動・制約

- **書き出しは確定した発話のみ**。喋っている途中の暫定テキストはファイルに出ない(数秒遅れて確定行が追記される)。暫定テキストはアプリのライブ表示でだけ見える。
- **自分/相手の区別は音声の経路による**。マイク=自分、システム音声=相手。同じ経路に複数人がいても、その中の誰かまでは区別しない。
- **システム音声(相手)は安定した署名がないと無音で失敗する**。相手側がまったく文字起こしされない場合は、`install.sh` による署名込みの導入ができているかを確認する。`hearcat status` の注意行にも出る。
- 言語は日本語(ja-JP)固定。文中の英語技術用語は拾えるが、精度は完全ではない。
- **疑問文には「？」が付く(実験的機能)**。語尾の形(「〜ですか」等)と、発話末尾のピッチ上昇(「大丈夫？」型)から推定する。イントネーションが平坦な疑問文は「。」のままのことがある。
- 録音を途中でオフ→オンすると、オフの間の音声はファイルに入らない(再生時間がその分短くなる)。

## トラブルシュート

ユーザーが「相手の声が文字起こしされない」「無音になる」「聞こえてるはずなのに transcript に相手の行が無い」等と言ったら:

1. `hearcat status` を実行し、「注意:」行が出ていないか確認する。出ていればその文言をそのまま案内し、原因が署名(システム音声キャプチャの安定署名)不足であることが多いと伝える。
2. リポジトリの `install.sh` を各マシンで実行し直すのが標準の直し方。dmg 配布版なら公証済みビルドの再インストールを案内する。
3. `hearcat start` で「HearCat.app を起動できません」と出た場合はアプリ未導入。設定パネルの「AI エージェント連携」の導入か、`install.sh` の実行を案内する。
4. マイク/音声認識の許可ダイアログが出ていないか確認する(初回のみ)。許可が降りていないと自分側の発話も欠落する。

## この skill が想定しない使い方

- 音声ファイル(.m4a) を AI が直接文字起こしする用途では**ない**。文字起こしはアプリ側の SpeechAnalyzer が担当し、AI は結果テキストを CLI 経由で読むだけ。
- ユーザーの Mac の外に音声・テキストを送る操作はしない(オンデバイス処理が本アプリの前提)。文字起こし内容をクラウド API へアップロードする類の依頼が来たら一度確認する。
