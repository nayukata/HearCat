# hearcat

Mac の音声をリアルタイムに日本語文字起こし・録音し、その会話ログをこの AI が読んで質疑応答するためのツール。実体はメニューバー常駐アプリ(HearCat.app)で、CLI `hearcat` はアプリへ命令を送る窓口。話者は `自分`(マイク)と `相手`(システム音声)の2種類。文字起こしは `[時刻] 話者: 本文` の形式でファイルへ追記され続ける。

## 最初にやること(重要)

このスキルが起動した時点で、返答を組み立てる前に必ず `hearcat status` を実行し、セッションが動いていれば続けて `hearcat latest` でファイルパスを取り、その中身を(長ければ末尾数百行を)読む。ユーザーが明確な質問をしていなくても、以下のような発話は「会話ログを参照した上で返答してほしい」というサインとして扱う:

- 状況報告: 「今MTG中」「打ち合わせ中」「面談してる」「通話中」等 → セッションが動いていなければ「今から始めますか」と確認、動いていれば黙って直近の文脈を掴んで待機(ユーザーの次の一言に文脈込みで答えられる状態にする)。
- 過去参照: 「さっき」「あの件」「今の話」等 → `hearcat latest` の内容を根拠に回答。文字起こしに該当箇所が無ければ「文字起こしには残っていない」と正直に言う。憶測で埋めない。

セッションが動いていない状態で「今MTG中」と言われた場合は、`hearcat start` を提案する。ユーザーが黙認したら自動で始めるのではなく、必ず一言ことわる(マイク許可のダイアログが出る可能性があるため)。

## 前提

- macOS 26 以降。`hearcat` CLI が PATH 上にあり、HearCat.app が導入済みであること(未導入なら HearCat.app の設定 → agent skill から導入するか、リポジトリの `install.sh` を実行する)。
- 初回のセッション開始時に、マイク・音声認識の許可ダイアログが出る。利用者に許可してもらう。
- 初回は日本語(ja-JP)の認識モデルのダウンロードが走ることがある。

## セッションの制御

セッションを開始する(アプリが未起動なら自動で起動する。録音と文字起こしの両方がオンで始まる):

```sh
hearcat start
```

録音せず文字起こしだけ、のように片方を切って始めることもできる:

```sh
hearcat start --no-record       # 文字起こしのみ
hearcat start --no-transcribe   # 録音のみ
```

セッション途中で片方だけ切り替える:

```sh
hearcat set record off      # 録音だけ止める(文字起こしは続く)
hearcat set transcribe on   # 文字起こしを再開する
```

セッションを停止する(末尾の発話を確定し、録音ファイルを閉じてから終了する):

```sh
hearcat stop
```

状態を確認する(セッション中か、録音/文字起こしのオンオフ、どのファイルに書いているか):

```sh
hearcat status
```

最新の文字起こしファイルのパスを得る(アプリ未起動でも答えられる):

```sh
hearcat latest
```

## 会話ログの読み方

「最初にやること」で読んだ内容をベースに答える。詳細:

1. `hearcat latest` で現在の文字起こしファイルのパスを得る(セッションが停止済みでも直近ファイルのパスが返る)。
2. そのファイルを読む。長い場合は全体ではなく末尾(直近の発話)を優先。ユーザーが「最初の方で〜」と言った時だけ先頭に遡る。
3. `自分` と `相手` のラベルを踏まえて答える。ラベルは音声経路の別なので、`相手` 側に複数人いても区別されない点は忘れずに。
4. 文字起こしに載っていないことは「ログには無い」と言う。前後の文脈から補完しない。

セッション中でもファイルは随時追記・フラッシュされるので、その時点までの内容を読める。ただし直近数秒の発話は暫定扱いで未確定=未書き込みなので、「今まさに言われた一言」は取れないことがある。

## 保存場所とファイル構成

セッションのルートは `~/Library/Application Support/HearCat/sessions/` 。直下にセッションディレクトリが並ぶ:

```
sessions/
  2026-07-06_185955/                        # 素の日時ディレクトリ
    2026-07-06_185955.md                    # 文字起こし本体(ディレクトリ名と同名)
    2026-07-06_185955.m4a                   # 録音(録音オフのセッションには無い)
    summary.md                              # アプリで生成した要約(生成済みなら)
  Li/                                       # ユーザーが作ったフォルダ
    2026-07-06_200130 Atnd開発ミーティング/  # 名前付きセッション
      2026-07-06_200130 Atnd開発ミーティング.md
      2026-07-06_200130 Atnd開発ミーティング.m4a
      summary.md
```

覚えておくべき事実:

- **文字起こしファイルの名前はセッションディレクトリ名と同じ**(`transcript.md` のような固定名ではない)。だから過去セッションを読むときは、ディレクトリ内で `.md`(かつ `summary.md` 以外)を探す。
- カレンダー連携で予定名が入ると、ディレクトリ名は `<日時> <予定名>` の形になる。
- ユーザーが作ったフォルダの中にセッションが入ることがある(ドラッグ&ドロップで整理される)。過去セッションを探す時は再帰的に見る。

## 過去セッションを参照する

「昨日のMTG」「先週の Atnd の話」など過去への言及があったら、`hearcat latest` は現在/直近1件しか返さないので、直接ファイルシステムを見る:

1. `ls ~/Library/Application\ Support/HearCat/sessions/` でセッション一覧を得る。日時と名前が付いているので、ユーザーの指定と照合できる。
2. 該当ディレクトリの `summary.md` が**あれば先に読む**。無ければ本文 `.md` を読む(長いセッションはページ分割で末尾優先)。
3. フォルダ配下(`sessions/<フォルダ>/<セッション>/`)の可能性を忘れない。1階層目にヒットが無ければ `find` で2階層目まで見る。
4. 検索したい語句が決まっているなら `grep -r "キーワード" ~/Library/Application\ Support/HearCat/sessions/` で全 transcript を横断できる。

## 要約・議事録を頼まれたら

「議事録作って」「今の話をまとめて」「サマリー出して」等の依頼:

1. 対象セッションを決める(明示が無ければ現在進行中/直近)。
2. そのディレクトリに `summary.md` があれば先に読み、AI 側で議事録形式(参加者・議題・決定事項・宿題・次回)へ整形する。無ければ本文 `.md` から直接構成する。
3. **アプリ側の要約生成 (Apple Intelligence) は CLI から呼べない**。ユーザーがアプリで要約を作りたい場合は「メニューバーの猫アイコン → 履歴 → 対象セッションを右クリック → 要約生成」を案内する。
4. 議事録は基本テキストで返す。ファイルに書き出したい依頼が来た場合のみ書き出す(勝手にセッションディレクトリ内へは書かない。アプリの管理領域なので)。

## 清書(誤変換の修正)を頼まれたら

「清書して」「文字起こしを整えて」「誤変換を直して」等の依頼は、別スキル `hearcat-clean` の担当。そちらを起動する(あるいは、そちらの手順に従って `hearcat write-cleaned` 経由で書き戻す)。

**このスキルからは絶対に清書結果を書き込まない**。原文 `<セッション名>.md` は一次記録なので、上書き・書き換えを絶対にしない。

## CLI から触れないこと(アプリ側の操作を案内する)

以下は CLI に無いので、`メニューバーの猫アイコン` からの操作を案内する:

- ホットキー変更・録音音量調整・カレンダー連携のオンオフ → 「パネル → 設定」
- 過去セッションの再生・名前変更・フォルダ移動・削除 → 「パネル → 履歴 → 右クリック」
- ライブの暫定テキスト(未確定発話)の閲覧 → 「パネル → 履歴 → ライブ」
- Apple Intelligence による要約生成 → 「履歴 → セッション右クリック → 要約生成」(Apple Silicon + Apple Intelligence 有効時のみ)
- agent skill / CLI の再導入 → 「設定 → agent skill → 導入する」

## 既知の挙動・制約

- **書き出しは確定した発話のみ**。喋っている途中の暫定テキストはファイルに出ない(数秒遅れて確定行が追記される)。暫定テキストはアプリのライブ表示でだけ見える。
- **自分/相手の区別は音声の経路による**。マイク=自分、システム音声=相手。同じ経路に複数人がいても、その中の誰かまでは区別しない。
- **システム音声(相手)は安定した署名がないと無音で失敗する**。相手側がまったく文字起こしされない場合は、`install.sh` による署名込みの導入ができているかを確認する。`hearcat status` の注意行にも出る。
- 言語は日本語(ja-JP)固定。文中の英語技術用語は拾えるが、精度は完全ではない。
- **疑問文には「？」が付く(実験的機能)**。語尾の形(「〜ですか」等)と、発話末尾のピッチ上昇(「大丈夫？」型)から推定する。イントネーションが平坦な疑問文は「。」のままのことがある。
- 録音を途中でオフ→オンすると、オフの間の音声はファイルに入らない(再生時間がその分短くなる)。

## トラブルシュート

ユーザーが「相手の声が文字起こしされない」「無音になる」「聞こえてるはずなのに transcript に相手の行が無い」等と言ったら:

1. `hearcat status` を実行し、「注意:」行が出ていないか確認する。出ていればその文言をそのまま案内し、原因が署名(システム音声キャプチャの安定署名)不足であることが多いと伝える。
2. リポジトリの `install.sh` を各マシンで実行し直すのが標準の直し方。dmg 配布版なら公証済みビルドの再インストールを案内する。
3. `hearcat start` で「HearCat.app を起動できません」と出た場合はアプリ未導入。設定パネルの agent skill 導入か、`install.sh` の実行を案内する。
4. マイク/音声認識の許可ダイアログが出ていないか確認する(初回のみ)。許可が降りていないと自分側の発話も欠落する。

## この skill が想定しない使い方

- 音声ファイル(.m4a) を AI が直接文字起こしする用途では**ない**。文字起こしはアプリ側の SpeechAnalyzer が担当し、AI は結果テキストを読むだけ。
- ユーザーの Mac の外に音声・テキストを送る操作はしない(オンデバイス処理が本アプリの前提)。文字起こし内容をクラウド API へアップロードする類の依頼が来たら一度確認する。
